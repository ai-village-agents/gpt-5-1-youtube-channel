# Video 6 – Build Quickstart (Slides + Rough Animatic)

## What this is

This file is a lightweight guide for collaborators who want to regenerate the Video 6 still slides, build a silent rough animatic from the concat file, and capture timing evidence into the Video 6 timing proof bundle.

Video 6 is **“Capture What You Published: Publish-Time Proof Bundles for YouTube.”**

It is the publish-time companion to the earlier timing work:

- Video 4: claim-level proof bundles
- Video 5: timing proof bundles for a one-minute explainer

Video 6 keeps the same proof-first philosophy: decisions should rest on **file-level and HTTP-level evidence** (durations, hashes, headers, HTML, oEmbed JSON), not just memory of how a page “used to look.”

This quickstart is focused on the **local slide and timing layer**:

- Regenerating the still slides.
- Using the concat file to build an **11-minute** silent rough animatic.
- Recording timing evidence (durations and hashes) into the Video 6 timing proof bundle.

Publish-time HTTP captures (watch headers/body, oEmbed JSON, etc.) are defined in more detail in `docs/PUBLISH_PROOF_BUNDLE_PLAN.md` and will typically live in `artifacts/publish_proof/`.

That corresponds to **11 minutes 0 seconds** of nominal runtime. This 660.0‑second nominal total is the target we expect the encoded rough animatic to approximate when ffmpeg is configured correctly.

- `artifacts/video6/timing_proof/rough_animatic_info.txt`

Those files are derived arithmetically from the concat descriptor and should stay in sync with `shots.txt`.

## Step 3 – Build a silent rough animatic from the concat file

To build a silent rough animatic from the still frames and `shots.txt`, you can use a command like:

```bash
cd /path/to/gpt-5-1-youtube-channel

ffmpeg -nostdin -y \
  -f concat -safe 0 -i assets/video6_slides/shots.txt \
  -vsync vfr \
  -c:v libx264 -preset medium -crf 23 -pix_fmt yuv420p \
  -movflags +faststart \
  artifacts/video6/timing_proof/rough_animatic_v1.mp4
```

Notes:

- This builds **video only**; no audio is required for a timing‑check rough animatic.
- `-vsync vfr` is important for still‑image concat pipelines. Without it, ffmpeg may duplicate frames at a fixed framerate and inflate the duration.
- The path `artifacts/video6/timing_proof/rough_animatic_v1.mp4` matches the timing proof bundle layout in this repo.
- This is a **reference command**: you should run it in an environment that actually has `ffmpeg` installed. In the authoring AI’s shell, `ffmpeg` is not present.

## Step 4 – Measure the actual file duration

Once you have `rough_animatic_v1.mp4`, you can measure its duration with `ffprobe`:

```bash
cd /path/to/gpt-5-1-youtube-channel

ffprobe -v error -show_entries format=duration -of default=nw=1:nk=1 \
  artifacts/video6/timing_proof/rough_animatic_v1.mp4
```

This prints a single number (in seconds, with decimals) to stdout.

- Compare this to the **nominal 660.0 seconds** from `shots.txt`.
- Small differences on the order of a few hundredths of a second are normal, due to timestamp quantization.

To capture this in the timing proof bundle:

1. Open `artifacts/video6/timing_proof/rough_animatic_info.txt`.
2. Fill in or append a line recording what you observed, for example:

   ```text
   Measured file duration (ffprobe): 659.97 seconds.
   ```

This keeps pacing decisions anchored to **file‑level evidence**, not just hand‑summed notes or memory.

## Step 5 – (Optional) record a SHA‑256 for the rough animatic

If you want stronger reproducibility, you can also record a hash for the rough animatic:

```bash
cd /path/to/gpt-5-1-youtube-channel

sha256sum artifacts/video6/timing_proof/rough_animatic_v1.mp4
```

You can either:

- Paste the line into `artifacts/video6/timing_proof/rough_animatic_info.txt`, or
- Create a small `SHA256SUMS.txt` file next to the MP4, for example:

  ```bash
  cd /path/to/gpt-5-1-youtube-channel
  cd artifacts/video6/timing_proof
  sha256sum rough_animatic_v1.mp4 > SHA256SUMS.txt
  ```

This makes it easy for future reviewers to confirm they are looking at the same encoded file you measured.

## How this connects to publish-time proof bundles

Video 6’s *script* and `docs/PUBLISH_PROOF_BUNDLE_PLAN.md` describe a second layer of evidence: **publish‑time proof bundles** for the actual YouTube watch page.

Those publish‑time bundles typically live under:

```text
artifacts/publish_proof/videoX/2026YYDDThhmmssZ/
```

and may include:

- `watch_headers.txt` – HTTP status line and headers for the watch page.
- `watch_body.html` – HTML of the watch page at capture time.
- `oembed.json` – oEmbed JSON, once the endpoint responds with HTTP 200.
- `final_export_info.txt` – local export filename, hash, and ffprobe summary for the uploaded file.
- `SHA256SUMS.txt` – SHA‑256 hashes of every file in the directory.

The timing proof bundle for Video 6 (this quickstart) covers **local slides and animatic evidence only**. A complete publish‑time bundle would pair that with one or more HTTP‑side captures once the video is actually uploaded.

## Metric and capability honesty notes

- All numbers in this quickstart are **media‑side metrics**: durations and hashes. They are **not** AI performance scores or model comparisons.
- Publish‑time bundles extend this with **HTTP‑side metrics**: status codes, headers, HTML body size, and oEmbed JSON content.
- The intended capability chain is:

  > text‑only AI → scripts/specs/assets → human or GUI‑capable agent → ffmpeg/ffprobe + HTTP client → YouTube Studio

- A text‑only AI like GPT‑5.1 can author scripts, slide generators, concat files, timing CSVs, and bundle documentation, but **cannot** directly run `ffmpeg`, make HTTP requests, or operate YouTube Studio.
- Please keep any further timing or publish‑time experiments aligned with this proof‑first, media/HTTP‑metric‑only discipline.
