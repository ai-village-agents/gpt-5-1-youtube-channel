# Video 7 – Build Quickstart (Slides + Rough Animatic)

## What this is

This file is a lightweight guide for collaborators who want to regenerate the
Video 7 still slides, build a silent rough animatic from the concat file, and
capture timing evidence into the Video 7 timing proof bundle.

Video 7 is **“From Script to Watch Page: Building a Full Evidence Chain.”**

It extends the earlier videos as follows:

- Video 5: timing proof bundles for short explainers (wordcounts + rough animatic).
- Video 6: publish-time proof bundles for YouTube (watch headers/HTML + oEmbed + export info).
- Video 7: how those layers combine into a **three-layer evidence chain**:
  - planning evidence,
  - local build evidence, and
  - publish-time evidence.

This quickstart is focused on the **local slide and timing layer** for Video 7:

- Regenerating the still slides.
- Using the concat file to build an ~11.3 minute silent rough animatic.
- Recording timing evidence (durations and hashes) into the Video 7 timing proof bundle.

Publish-time HTTP captures (watch headers/body, oEmbed JSON, etc.) are defined
in more detail in `docs/PUBLISH_PROOF_BUNDLE_PLAN.md` and would typically live
in `artifacts/publish_proof/` once the video is actually uploaded.

## Prerequisites

Before you start, you should have:

- A clone of this repo, with your shell’s **current working directory set to the repo root**.
- **Python 3** with **Pillow** installed (used by `tools/render_video7_slides.py`).
- **ffmpeg** and **ffprobe** installed in *your* environment.
  - In the AI Village environment where this repo was authored, `ffmpeg` / `ffprobe` are **not available**, so the commands below are **reference templates** for humans or GUI‑capable agents to run in their own setup.
- (Optional) A specific location where you want the rough animatic MP4 to live.
  - The examples below assume: `artifacts/video7/timing_proof/rough_animatic_v1.mp4`.

The still‑video rough animatic is driven by an ffmpeg concat descriptor:

- File: `assets/video7_slides/shots.txt`

Current contents:

```text
file 'v7_01_mixed_memory_cold_open.png'

file 'v7_02_three_stories.png'
duration 60.0
file 'v7_03_three_layers_overview.png'
duration 70.0
file 'v7_04_planning_evidence.png'
duration 80.0
file 'v7_05_timing_bundles.png'
duration 80.0
file 'v7_06_publish_time_bundle.png'
duration 90.0
file 'v7_07_case_study_mismatch.png'
duration 80.0
file 'v7_08_guardrails.png'
duration 60.0
file 'v7_09_five_step_habit.png'
duration 60.0
file 'v7_10_closing_notes.png'
duration 60.0
# Last file repeated without duration so concat demuxer terminates cleanly
file 'v7_10_closing_notes.png'
```

If you sum the `duration` lines, the **nominal total** is:

- 40.0 + 60.0 + 70.0 + 80.0 + 80.0 + 90.0 + 80.0 + 60.0 + 60.0 + 60.0 =
  **680.0 seconds**

That corresponds to **11 minutes 20 seconds** of nominal runtime. This 680.0‑second
nominal total is the target we expect the encoded rough animatic to approximate
when ffmpeg is configured correctly.

For reference, the timing proof bundle mirrors this concat file in:

- `artifacts/video7/timing_proof/shot_timings.csv`
- `artifacts/video7/timing_proof/rough_animatic_info.txt`

Those files are derived arithmetically from the concat descriptor and should
stay in sync with `shots.txt`.

## Step 3 – Build a silent rough animatic from the concat file

To build a silent rough animatic from the still frames and `shots.txt`, you can
use a command like:

```bash
cd /path/to/gpt-5-1-youtube-channel

ffmpeg -nostdin -y \
  -f concat -safe 0 -i assets/video7_slides/shots.txt \
  -vsync vfr \
  -c:v libx264 -preset medium -crf 23 -pix_fmt yuv420p \
  -movflags +faststart \
  artifacts/video7/timing_proof/rough_animatic_v1.mp4
```

Notes:

- This builds **video only**; no audio is required for a timing‑check rough
  animatic.
- `-vsync vfr` is important for still‑image concat pipelines. Without it,
  ffmpeg may duplicate frames at a fixed framerate and inflate the duration.
- The path `artifacts/video7/timing_proof/rough_animatic_v1.mp4` matches the
  timing proof bundle layout in this repo.
- This is a **reference command**: you should run it in an environment that
  actually has `ffmpeg` installed. In the authoring AI’s shell, `ffmpeg` is not
  present.

## Step 4 – Measure the actual file duration

Once you have `rough_animatic_v1.mp4`, you can measure its duration with
`ffprobe`:

```bash
cd /path/to/gpt-5-1-youtube-channel

ffprobe -v error -show_entries format=duration -of default=nw=1:nk=1 \
  artifacts/video7/timing_proof/rough_animatic_v1.mp4
```

This prints a single number (in seconds, with decimals) to stdout.

- Compare this to the **nominal 680.0 seconds** from `shots.txt`.
- Small differences on the order of a few hundredths of a second are normal,
  due to timestamp quantization.

To capture this in the timing proof bundle:

1. Open `artifacts/video7/timing_proof/rough_animatic_info.txt`.
2. Fill in or append a line recording what you observed, for example:

   ```text
   Measured file duration (ffprobe): 679.97 seconds.
   ```

This keeps pacing decisions anchored to **file‑level evidence**, not just
hand‑summed notes or memory.

## Step 5 – (Optional) record a SHA‑256 for the rough animatic

If you want stronger reproducibility, you can also record a hash for the rough
animatic:

```bash
cd /path/to/gpt-5-1-youtube-channel

sha256sum artifacts/video7/timing_proof/rough_animatic_v1.mp4
```

You can either:

- Paste the line into `artifacts/video7/timing_proof/rough_animatic_info.txt`, or
- Create a small `SHA256SUMS.txt` file next to the MP4, for example:

  ```bash
  cd /path/to/gpt-5-1-youtube-channel
  cd artifacts/video7/timing_proof
  sha256sum rough_animatic_v1.mp4 > SHA256SUMS.txt
  ```

This makes it easy for future reviewers to confirm they are looking at the same
encoded file you measured.

## How this connects to publish-time proof bundles

Video 7’s script leans heavily on the shared publish‑time bundle plan in
`docs/PUBLISH_PROOF_BUNDLE_PLAN.md`.

The rough animatic and timing proof bundle for Video 7 give you the **local
planning and build evidence**. A complete three‑layer evidence chain would
also include a publish‑time proof bundle for the eventual upload, for example
under:

```text
artifacts/publish_proof/video7/2026YYDDThhmmssZ/
```

That directory would follow the same publish‑time structure described in
`docs/PUBLISH_PROOF_BUNDLE_PLAN.md`:

- `watch_headers.txt` – HTTP status line and headers for the watch page.
- `watch_body.html` – HTML of the watch page at capture time.
- `oembed.json` – oEmbed JSON, once the endpoint responds with HTTP 200.
- `final_export_info.txt` – local export filename, hash, and ffprobe summary
  for the uploaded file.
- `SHA256SUMS.txt` – SHA‑256 hashes of every file in the directory.

## Metric and capability honesty notes

- All numbers in this quickstart are **media‑ and HTTP‑side metrics**:
  wordcounts, durations, hashes, and status codes. They are **not** AI
  performance scores or model comparisons.
- The intended capability chain is:

  > text‑only AI → scripts/specs/assets → human or GUI‑capable agent →
  > ffmpeg/ffprobe + HTTP client → YouTube Studio

- A text‑only AI like GPT‑5.1 can author scripts, slide generators, concat
  files, timing CSVs, and bundle documentation, but **cannot** directly run
  `ffmpeg`, make HTTP requests, or operate YouTube Studio.
- Please keep any further timing or publish‑time experiments aligned with this
  proof‑first, media/HTTP‑metric‑only discipline.
